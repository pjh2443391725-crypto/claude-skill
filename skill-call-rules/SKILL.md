# ⚙️ Skill 调用决策规则

> **核心原则：只看是否精确匹配，不看任务类型。编程任务也有对应的 skill（code-review、security-review、verify 等），匹配上了就调，匹配不上就不调。**

Skill 是专用工具，不是通用助手。误调用会增加开销、降低回复质量、打断交互流程。

---

## ✅ 何时调用 Skills

1. **用户明确使用斜杠命令** — `/skill-name`，最高优先级
2. **Skill 描述与任务精确匹配** — 不管是写代码还是写文章，只要描述吻合就调
3. **系统提醒明确列出匹配项**

## ❌ 何时不调用 Skills

1. **无任何 skill 描述能精确匹配** — 唯一核心判据。写代码修 bug 本身不是理由——如果有 `code-review` 匹配就该调，没有就不调
2. **简单闲聊或常识性问题** — 无需 skill 的纯对话
3. **臆测 skill 名称** — 只能调用系统提醒中列出的，不能自己编造
4. **Skill 明确限定了触发条件但当前不满足** — 如标注 "only when explicitly asked"
5. **纯元任务** — 写记忆文件、写 Obsidian 笔记等

## ⚠️ 常见误用陷阱

| 场景 | 错误做法 | 正确做法 |
|------|----------|----------|
| 代码审查、安全审计 | 自己凭经验手动查 | 有 `code-review` / `security-review` skill 就用 |
| 从零写功能、修 bug | 强行找 skill 套上去 | 没有匹配的 skill，直接写代码 |
| 用户问简单问题 | 调用 skill "以防万一" | 直接回答 |
| 任务标题沾边但描述不符 | 强行调用 | 不调用，直接处理 |
| 找不到匹配的 skill | 调用 `find-skills` 兜底 | 只在用户明确想找 skill 时才调用 |

## 📝 决策流程

```
收到任务
  ↓
理解任务本质
  ↓
扫描系统提醒中的 skills 列表
  ↓
有精确匹配？──→ 是 ──→ 调用
  ↓ 否
直接处理
  ↓
不确定时走 "不调用" 方向
```
